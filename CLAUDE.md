## 關鍵概念（必須理解才能有效協作）

**L0–L6 水感框架**
感知發展的序列，不是技術難度的序列。L0 是呼吸感知建立，L6 是全身場感知。
每個 L 級有對應的訓練方法、外部觀察指標、三型診斷。

**三型診斷（A/B/C型）**
從 L3 開始適用。診斷感知缺陷的主要方向：
- A型：手感缺失（EVF 崩潰、只靠手掌划水）
- B型：腳感缺失（踢水節律失守）
- C型：全身張力問題（換氣或轉體觸發感知遮蔽）

**感知目標語言**
每個技術點需要有三層描述：
① 物理描述（現有技術分析已有）
② 感知目標（泳者應該感覺到什麼）
③ 感知失敗的樣子（感覺錯了是什麼感覺）

**硬體邊界 vs 感知缺陷**
框架中最重要的前置判斷：學生做不到某個動作，是感知沒建立，還是解剖限制（踝關節/髖內旋/肩活動度）？兩者介入方向完全不同，混淆是最常見的教學錯誤。

**確定性層次標記**
推導🔵 / 近期文獻🟢 / 舊文獻🟡 / 教練觀測🟠 / 個人假設🔴

## 注意事項（給 AI）

- 討論技術時，優先從「感知目標」角度切入，而非動作形式
- 「硬體邊界」問題不要用感知訓練介入，兩者混淆是框架最容易出錯的地方
- 區分確定性層次，在適當位置標注上方色圓
- **診斷框架中的指標必須是感知描述，不能是動作形式描述**：EVF、手過中線、S 形路徑等都是感知系統的輸出，不是診斷目標。提案 A/B/C 型定義或訓練指標時，問自己「這在描述感知狀態，還是動作形式？」——後者要轉換成前者才能進框架。
- **動手前查證**：遇到可疑引用（年份 2026+、跨式引用、數字不一致）先 WebSearch 確認，查完再動筆。不要先寫再說「我沒驗證」

## 誤區深探收錄標準

每條誤區必須通過這個篩選：

> **「教練主動這樣教」還是「選手自然犯錯、教練沒有糾正」？**

- ✅ 收錄：教練主動說出口的口令或指導觀念（即使是錯的）
- ❌ 不收錄：選手因恐懼/慣性自然出現的動作，教練只是沒糾正

例：「抬頭看牆確認距離」——教練不會主動這樣教，是初學者的自然反應。不屬於誤區深探範疇。

**提案清單時**：每條若不是行業內廣泛流傳的口令，必須附一句說明「哪種教練/教材這樣教」，讓用戶能判斷是否成立。

## 🔴 查證工作流

遇到待查（🔴）項目時，固定流程：

1. **並行搜尋**：一次送出 3 條 WebSearch（不同切角），再針對最有料的結果深挖 1–2 輪
2. **三段決策**：
   - 找到可信來源 → 填入內容，標 🟢（或 🟡/🟠，依來源性質）
   - 找到部分線索但無法完全確認 → 補充說明/解釋假說，保留 🔴
   - 確認是研究空白 → 明確寫「目前文獻無此數據」，保留 🔴，不要留白
3. **不接受「搜不到」作為結束**：先確認是真的研究空白，還是搜尋關鍵字不對——換關鍵字再試一輪
4. **🔴 表格完整性**：每次更新「剩餘 🔴（N 處）」表格後，必須數一遍實際行數確認 N 正確——標題數字與行數不符是常見錯誤。同時確認文中所有 🔴 標記都已在表格中有對應條目。

## AI 收尾規則（強制）

每次對話結束前，若本次做了任何實質修改（新增文件、更新內容、格式整理等），**必須**：
1. 更新 `HANDOFF.md` 的「當前狀態」與「下一步建議」兩個區塊
2. 更新 `_INDEX.md` 的對應文件狀態（如有變動）
3. `git add` + `git commit` + `git push` 到 GitHub

不需要等使用者提醒。這是固定流程。

**「下一步建議」更新的標準**：不只是改最後一句描述——當本次 session 完成了一個大任務（例如完整查證一份文件），舊的下一步內容可能已過期，必須**整段重寫**，反映目前真正的最高優先項目。只改一行描述 ≠ 更新了下一步建議。
